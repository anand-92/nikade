# openOwl v1.0.1 Release Notes

**Release Date**: 2026-03-18

Significant quality-of-life update: terminal search, menu bar shortcut discoverability, Claude status monitoring, visual polish, and a batch of bug fixes.

---

## New Features

### Terminal Search (Cmd+F)

Full-text search is now available inside every terminal pane.

- Press **Cmd+F** to open the search bar; it sits above the terminal content in the layout (VStack), not as a floating overlay ‚Äî terminal height auto-adjusts
- **Return** / **Shift+Return** navigate to next / previous match; handled in `AppDelegate.handleLocalKeyDown` before the Command-key guard so they work without modifier keys
- **Esc** closes the search and returns focus to the terminal
- Each pane maintains independent search state ‚Äî searching in one pane doesn't affect others
- Search bar auto-closes when you switch focus to another pane

### Claude Status Incident Banner

When Anthropic reports an active incident on [status.claude.com](https://status.claude.com), a dismissible warning card appears at the bottom of the sidebar.

- Polls `status.claude.com/history.rss` every 2 minutes in the background
- Shows only when an unresolved incident exists ‚Äî disappears automatically once resolved
- Dismiss button marks the current incident as seen; it won't reappear for the same incident
- "Open status page" link opens the incident in the browser
- Dismissed incident IDs are persisted across app restarts

## Improvements

### Sidebar Split-Pane Rows

The per-pane status rows in the sidebar are now larger and easier to interact with.

- Font size: 10 ‚Üí 11 pt
- Status dot: 5 ‚Üí 7 px
- Vertical padding: 1 ‚Üí 4 pt (meets macOS HIG minimum click target)
- Hover highlight using `.quaternary` background
- Clicking a row now focuses the corresponding terminal pane
- Full accessibility support (`.isButton` trait, VoiceOver label + hint describing bell state beyond color)

### Visual: Liquid Glass Effects (macOS 26+)

On macOS 26 (Tahoe), two UI elements now use the Liquid Glass material:

- **Editor tab bar** ‚Äî selected tab uses tinted glass (accent color tint)
- **Deployment action buttons** (Start / Stop / Restart) ‚Äî filled glass with accent tint

Falls back gracefully on macOS 14‚Äì25 with no visual change.

### Settings Window (Cmd+,)

New app-level Settings with two options:

- **Appearance** ‚Äî System / Light / Dark. Controls the window chrome (sidebar, tab bar, etc.) independently from the terminal. Defaults to System (follows macOS).
- **Terminal Theme** ‚Äî Searchable picker with 463 built-in themes (sourced from [iterm2-color-schemes](https://github.com/mbadolato/iTerm2-Color-Schemes)). Default is Tomorrow Night (ghostty's built-in palette). Theme change requires app restart.

Settings are stored in `~/Library/Application Support/com.openowl.app/config` using ghostty-compatible config format ‚Äî users can also edit this file directly with any ghostty configuration key.

**Note:** openOwl's terminal config is fully isolated from the Ghostty app. Installing or configuring Ghostty does not affect openOwl and vice versa.

### Debug Build Icon

Debug builds now display a distinct app icon with a red "DEV" banner in the Dock, making it easy to distinguish from Release builds at a glance. Implemented via `NSApp.applicationIconImage` at runtime (works around an Xcode asset catalog limitation where non-primary icon sets generate incomplete `.icns` files).

### Light / Dark Mode Color Consistency

All hardcoded dark-mode RGB colors in the app palette have been replaced with macOS semantic system colors (`windowBackgroundColor`, `controlBackgroundColor`, `separatorColor`, etc.). The app now adapts correctly to both Appearance modes without any per-mode overrides.

### Performance

- `ISO8601DateFormatter` is now a static singleton (previously allocated once per Git commit row) ‚Äî eliminates thousands of allocations on large repos
- Commit list switched to `LazyVStack` ‚Äî only visible commits are rendered; previously the full list was laid out at once
- Graph layout computation is cached in `@State` + `onChange`; no longer recalculates on every view update
- File content loading in the editor is now fully async (`Task.detached`) ‚Äî avoids main-thread stall on large files

## Bug Fixes

### Terminal Split-Pane Drag Reliability

Pane drag-and-drop now works consistently in all scenarios.

- **Root cause 1**: `TerminalScrollView` was intercepting drag events because `NSItemProvider(object: NSString)` produces `public.utf8-plain-text`, which matched the terminal's accepted drop types. Fixed by using a private UTType `com.openowl.terminal.pane-drag` that the terminal scroll view ignores.
- **Root cause 2**: The drop overlay condition `draggingPaneID != paneID` evaluated to `true` when `draggingPaneID` was `nil` (Swift: `nil != UUID()` is always `true`), causing the `Color.clear.contentShape(Rectangle())` overlay to permanently block mouse events. Fixed by using `let draggingID = workspace.draggingPaneID, draggingID != paneID` to properly unwrap the Optional.

### Terminal Search ‚Äî Event Blocking

Fixed three independent issues where the search overlay blocked terminal mouse and keyboard events:

- **Mouse events blocked (selection, copy)**: A SwiftUI-level `.contentShape(Rectangle())` + `.onDrop(of: [UTType.fileURL])` on the pane container made the entire area a SwiftUI hit target, intercepting all mouse events before they reached the terminal `NSView`. Removed the redundant SwiftUI drop handler ‚Äî file drops are already handled at the AppKit level by `TerminalScrollView`.
- **Enter key blocked (IME confirmation)**: SwiftUI's `@FocusState` and AppKit's `firstResponder` fall out of sync in hybrid apps. After clicking the terminal, AppKit knew the terminal had focus but SwiftUI still believed the search `TextField` was focused, causing `.onKeyPress(.return)` to fire and return `.handled`. Replaced with `.onSubmit`, which only triggers when the `TextField` is the active text input.
- **Search bar buttons unclickable**: The search overlay was rendered underneath the pane drop delegate overlay in the z-order. Moved the search overlay to the outermost position.

### File Explorer Crash (MinimapView)

Fixed a crash that occurred when opening files with certain editor themes.

- **Root cause**: `MinimapView.setTheme()` called `.brightnessComponent` directly on semantic catalog colors (e.g. `controlBackgroundColor`), which return an undefined value until converted to a concrete color space. Added `.usingColorSpace(.deviceRGB)?` guard before accessing `brightnessComponent`.
- **Dependency**: Patched in a fork of `CodeEditSourceEditor` (`sanvibyfish/CodeEditSourceEditor`, branch `fix/minimap-brightness-catalog-color`).

### Cmd+Arrow ‚Äî Single-Pane Conflict

`Cmd+‚Üê/‚Üí/‚Üë/‚Üì` no longer interfere with ghostty's own bindings when only one pane is open. Arrow-key pane navigation is now intercepted only when two or more panes exist in the active tab.

### File Explorer ‚Äî State Mutation During View Update

Fixed 7 SwiftUI runtime warnings ("Modifying state during view update, this will cause undefined behavior") across `FileExplorerView` and `OutlineTreeViewController`.

- **Root cause**: `updateNSViewController` ‚Üí `updateData` ‚Üí `reloadData()` ‚Üí `restoreSelectedState` ‚Üí `outlineView.selectRowIndexes` ‚Üí `outlineViewSelectionDidChange` ‚Üí `onSelectFile` ‚Äî all executing synchronously within SwiftUI's render cycle. Fixed with `isProgrammaticSelection` flag wrapping the entire `reloadData()` + restore block in `updateData`, plus `Task { @MainActor in ‚Ä¶ }` for `onChange(of: store.selectedNodeID)`.

### File Explorer ‚Äî Expand Directory Missing Git Status

Lazily-expanded directories now display correct git status badges (A/M/D indicators).

- **Root cause**: `expandDirectory` used `gitContext: .empty`, so all children got `gitState: nil`. Fixed by caching the latest `GitContext` after each full scan and using it in `expandDirectory` for both `shouldIgnore` filtering and git state assignment. Sub-directories infer their state by scanning `statusByAbsolutePath` entries with a matching path prefix.

### File Explorer ‚Äî Expand Shows Empty / Delayed Children

Expanding a directory via arrow keys now shows children immediately.

- **Root cause**: `outlineViewItemWillExpand` called `store.expandDirectory` which updated the store's `nodeIndex`, but the controller's local copy was only synced via `updateNSViewController` (async, next render cycle). When `NSOutlineView` immediately queried the data source, it got 0 children. Fixed by adding `syncData(rootNodes:nodeIndex:)` to `OutlineTreeViewController` and calling it synchronously from the `onExpandDirectory` callback wrapper.

### File Explorer ‚Äî Refresh Collapses Expanded Directories

File system changes no longer cause the tree to collapse and re-expand.

- **Root cause**: `FileWatcher` ‚Üí `refreshNow()` ‚Üí `refresh()` included a shallow scan phase (`maxDepth: 1`) that replaced the fully-expanded tree with a flat one. Fixed by routing subsequent refreshes (when `rootNodes` is non-empty) through `refreshFullOnly()`, which skips the shallow phase entirely.

### File Explorer ‚Äî Git Badges Clipped on Initial Load

Git status badges no longer appear off-screen when the file explorer first loads.

- **Root cause**: `autoresizesOutlineColumn = true` expanded the column beyond the clip view width during the `@Observable` ‚Üí `updateNSViewController` render cycle (before the view had its final frame). Unlike the previous Combine-based async subscription, `updateNSViewController` runs synchronously during SwiftUI's render cycle. Fixed by setting `autoresizesOutlineColumn = false` and adding a conditional `updateData` guard (`controller.rootNodes != store.rootNodes`) to skip redundant reloads.

### Quick Open (Cmd+P) ‚Äî Search Field Shortcut Passthrough

Cmd+F and other standard editing shortcuts now work correctly inside text fields (search field, commit message, etc.). Previously, the local key monitor was intercepting these shortcuts even when a non-terminal view had focus. Fixed by guarding terminal shortcuts on `firstResponder is TerminalNSView`.

### Terminal Search ‚Äî Auto-Close Stole Focus From New Pane

Switching terminal panes while a search bar was open no longer bounces focus back to the previous pane.

- **Root cause**: `closeSearch()` always called `refocusTerminal()`, including the auto-close path triggered by `onChange(of: isFocused)`. Refocusing the source pane fought the new pane's focus acquisition, requiring an extra click. Fixed with `closeSearch(refocus: Bool = true)` ‚Äî the auto-close path passes `refocus: false`.

### Pane Drag ‚Äî Cancelled Drag Left Panes Unresponsive

Cancelling a pane drag (dropping outside any valid target) no longer leaves non-source panes stuck behind an invisible hit-test overlay.

- **Root cause**: `draggingPaneID` was only cleared inside `PaneDropDelegate.performDrop()`. If the user cancelled the drag (drop outside all panes, or back on the source), `performDrop` was never called and `draggingPaneID` remained set. The conditional `contentShape(Rectangle())` overlay persisted on every non-source pane, permanently blocking mouse events until a later successful drop. Fixed by monitoring `leftMouseUp` in `AppDelegate` ‚Äî after a brief async yield (to let `performDrop` run first for successful drops), `cancelDragIfActive()` clears residual state.

### File Explorer ‚Äî Git Errors Silently Poisoned Context

When `git status` or `git ls-files --ignored` failed (binary missing, corrupt repo, lock conflict), the error was silently swallowed and an empty `GitContext` replaced the previous valid one. All subsequent lazy directory expansions then showed no git badges and surfaced gitignored paths as visible entries.

- **Root cause**: `repoRoot(for:)`, `loadIgnoreContext`, and `loadGitStatus` all used `try?`, collapsing every error ‚Äî including transient `commandFailed` ‚Äî into a silent empty result indistinguishable from "not a git repo". Fixed with `do/catch` in all three; `GitServiceError.notGitRepository` returns empty silently (expected); every other error is logged via `NSLog`.

### File Explorer ‚Äî Directory Entry Silently Misclassified as File

A directory that failed `resourceValues` lookup during lazy expand was silently treated as a file ‚Äî no expand arrow, no children.

- **Root cause**: `try? url.resourceValues(...)` returned `nil` on permission errors, symlink loops, or race-condition deletions, causing `isDir = false`. Instead of skipping the entry, the code proceeded to create a leaf node. Fixed by replacing `try?` with explicit `do/catch`: file-not-found is skipped silently; all other errors are logged and the entry is skipped.

### File Explorer ‚Äî Delete Silently Failed, File Reappeared

When `Trash Item` failed (permission denied, locked file), the file disappeared from the UI but remained on disk. The `FileWatcher` then triggered a rescan and the item reappeared with no explanation.

- **Root cause**: `DispatchQueue.global + try?` discarded all `FileManager.trashItem` errors. Fixed by switching to `Task` for structured concurrency; failed items surface via `errorMessage` and `refreshNow()` restores them in the tree.

## Technical Details

- **Platform**: macOS (Apple Silicon + Intel)
- **Terminal engine**: libghostty with Metal GPU rendering
- **Language**: Swift + SwiftUI/AppKit hybrid
- **Min macOS**: 14.0 (Sonoma)
- **Architecture**: All 7 observable stores migrated from `ObservableObject` + Combine to Swift `@Observable` (macOS 14+), eliminating `@Published`, `AnyCancellable`, and redundant Combine subscriptions
