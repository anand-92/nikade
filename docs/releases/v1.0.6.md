# openOwl v1.0.6 Release Notes

**Release Date**: 2026-04-24

Terminal quality-of-life pass: multi-file drag fixed, scrollbar becomes draggable, and heavy TUI output no longer saturates the main thread. Plus: tab switches no longer blow away File Explorer editor tabs and Git commit drafts.

---

## Terminal Fixes

### Multi-File Drag Only Ingested the First File

Dragging multiple files from Finder into the terminal produced a single path instead of all of them. Root cause: `performDragOperation` checked `pb.string(forType: .URL)` first, which on a multi-file drag returns only the first file's URL as a plain string. The multi-value branch (`readObjects(forClasses: [NSURL.self])`) was never reached.

Finder places both representations on the pasteboard: `.URL` (single, first only) *and* `.fileURL` (the full list, readable via `readObjects`). Whoever reads first wins.

**Fix:** Check `.fileURL` via `readObjects([NSURL.self])` **first**, fall back to `.URL` only for non-file URLs (e.g. `http://`), then `.string`. This mirrors the logic already used by `pasteFromClipboard` — the two code paths are now consistent.

### Scrollbar Not Draggable

The scrollbar rendered the correct position from `GHOSTTY_ACTION_SCROLLBAR` but ignored mouse input. `ScrollIndicatorView` only implemented `draw(_:)` — no `mouseDown` / `mouseDragged` / `mouseUp`, no hit testing. Users could only scroll via wheel/trackpad.

**Fix:** Made `ScrollIndicatorView` interactive:

- **Knob drag**: `mouseDown` on the knob records the cursor-to-knob offset; `mouseDragged` preserves that offset (knob doesn't snap to cursor — matches native NSScroller feel).
- **Track click**: `mouseDown` on empty track centers the knob under the cursor, then enters drag mode anchored there.
- **Position mapping**: drag position `∈ [0, 1]` → row via `row = round(position × (state.total - state.len))` → forwarded to ghostty via the `scroll_to_row:N` binding action.
- **Hit-test gate**: `hitTest` returns nil when `alphaValue < 0.01` or `proportion >= 1`, so clicks pass through to the terminal when the indicator is hidden or there's no scrollback to scroll.
- **Fade suspended during drag**: `onInteractionChange` callback tells `TerminalScrollView` to invalidate the fade-out timer while the user is actively dragging; fade resumes 1s after `mouseUp`.

## Performance

### Main-Thread Saturation During Heavy TUI Output

Running `cat large.log`, `yes | head -100000`, or redrawing-heavy TUIs (htop, neovim scroll) caused UI input lag and occasional stutter. Root cause: `wakeup_cb` (called by libghostty from any thread whenever new data arrives) did a bare `DispatchQueue.main.async { manager.tick() }` every time. Under sustained output, hundreds of `tick()` blocks piled up on the main queue, delaying user events (clicks, scroll, keystrokes).

The existing code had a comment acknowledging this: *"There is probably some improvements we can make to coalesce multiple ticks."*

**Fix:** Added `scheduleTick()` — a coalescing wrapper:

```swift
func scheduleTick() {
    os_unfair_lock_lock(&tickLock)
    if tickScheduled { os_unfair_lock_unlock(&tickLock); return }
    tickScheduled = true
    os_unfair_lock_unlock(&tickLock)

    DispatchQueue.main.async { [weak self] in
        // Clear flag BEFORE tick so a wakeup fired during tick still gets scheduled.
        os_unfair_lock_lock(&self.tickLock)
        self.tickScheduled = false
        os_unfair_lock_unlock(&self.tickLock)
        self.tick()
    }
}
```

Properties:
- N wakeups → at most 1 pending main-queue block.
- `os_unfair_lock` (nanosecond-scale) chosen over `NSLock` because `wakeup_cb` is a hot path.
- Flag cleared *before* `tick()` runs, so if libghostty wakes up mid-tick, the next wakeup re-schedules correctly (no lost events).
- `ghostty_app_tick` is already batch-idempotent, so coalescing is safe: one tick processes all pending events.

Normal typing is unchanged. The win is visible only under sustained data bursts.

## UI Fixes

### Tab Switch Wiped File Explorer Editor Tabs

Opening files in File Explorer, switching to the Terminal tab, then switching back produced an empty editor — all previously-open files were gone. Same problem affected the Git Changes commit-message draft, expanded hunks, and scroll position.

Root cause: `ContentView` rendered three of the four top-level tabs via `if activeTab == .xxx { … }`. When the condition flipped, SwiftUI destroyed the entire view tree, including every `@State` inside (`openTabs`, `activeTabURL`, `tabStorages`, `dirtyTabs`, `editorState`, etc.). The Terminal tab didn't have this bug because it used `.opacity(0)` + `.allowsHitTesting(false)` to stay mounted.

The two patterns diverged historically: Terminal was forced into "stay mounted" to preserve its Metal surface, while File Explorer / Git Changes / Deployments were added later without matching that convention.

**Fix:** All four tabs now use the opacity-based pattern. The inactive tabs remain in the view tree (alpha 0, hit testing off), so their `@State` survives the user moving between tabs.

**Companion fix — conditional keyboard shortcuts:** With all views mounted at once, shortcuts registered inside them (`Cmd+S`, `Cmd+W`, `Cmd+Return`) would have fired from inside the Terminal tab. Each shortcut is now gated on `activeTab`:

- `FileExplorerView`: `Cmd+S` (save) and `Cmd+W` (close tab) only register when `activeTab == .fileExplorer`. The duplicate `Cmd+P` in the tree panel header was removed entirely — the global one in `ContentView` already covers it (leaving both would produce "ambiguous shortcut" warnings).
- `GitChangesView`: `Cmd+Return` (commit) moved off the visible button into a conditional hidden `.background` button that only registers when `activeTab == .gitChanges && commitEnabled`.
- `DeploymentPanelView`: no shortcuts, nothing to gate.
