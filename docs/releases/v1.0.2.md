# openOwl v1.0.2 Release Notes

**Release Date**: 2026-03-18

Bug fix release + terminal improvements: surface retention across project switches, command-finish notifications, and configurable alert sounds.

---

## Bug Fixes

### Deployment — 100% CPU on Continuous Log Output

Running a deployment that continuously outputs logs (e.g. Node.js dev server with verbose logging) caused the app to spike to 100% CPU and become unresponsive.

Three independent root causes were identified and fixed:

- **Root cause 1 — `readabilityHandler` EOF spin loop** (`DeploymentProcessManager.swift`, `DeploymentStore.runBuild`): Foundation's `readabilityHandler` continues to be called after the pipe reaches EOF. The existing code `guard !data.isEmpty else { return }` returned early but never nil'd out the handler, so the system immediately re-invoked it — creating a tight busy loop between process exit and `terminationHandler`. Fixed by setting `handle.readabilityHandler = nil` when `availableData` returns empty data.

- **Root cause 2 — Unthrottled `@Observable` mutations** (`DeploymentStore.appendLog`): Every chunk of process output dispatched to the main thread and directly mutated `logContent` (an `@Observable` property), triggering a SwiftUI view re-render per chunk. High-throughput processes could cause hundreds of re-renders per second. Fixed by buffering incoming text in a non-observed `logBuffer` and flushing to `logContent` at most every 200ms via a coalescing timer.

- **Root cause 3 — Duplicate log updates** (`DeploymentStore.loadLog` + `startProcess`): When a running deployment was selected, both the real-time `onOutput` callback and the 0.5s file poll timer were actively appending the same data to `logContent` — doubling the write volume and SwiftUI re-render frequency. Fixed by tracking active real-time streams in `activeStreamIDs` and skipping file polling when a stream is active.

### Deployment — Log Auto-Scroll Animation Overhead

The log panel's auto-scroll-to-bottom was wrapped in `withAnimation`, creating animation transactions on every `logContent` change. Combined with high-frequency updates, this added significant overhead to the main thread. Fixed by removing the `withAnimation` wrapper — the scroll still tracks the bottom, but without animation cost.

### Terminal — File Drag-Drop Sent to Wrong Project

Dragging a file from Finder into the terminal caused the file path to appear in the **previous project's terminal** instead of the current one, and the active terminal went blank.

- **Root cause**: When switching project tabs, old terminal NSViews remain in the AppKit view hierarchy hidden via SwiftUI's `.opacity(0)`. AppKit's drag-and-drop system does not respect SwiftUI opacity — it still routes drag events to invisible NSViews in the ZStack. The first (hidden) view would accept the drag, write the file path to its own ghostty surface, and steal first responder via `window?.makeFirstResponder(self)`.
- **Fix**: Added `isEffectivelyVisible` check (walks the superview chain for `isHidden` or `alphaValue < 0.01`) to all drag handlers in `TerminalScrollView` (`draggingEntered`, `draggingUpdated`, `prepareForDragOperation`, `performDragOperation`). Hidden views now reject drags, letting AppKit route the event to the visible terminal.

### Sidebar — Branch Click Navigated to Blank Page

Clicking a branch or split-pane row in the sidebar activated the project but did not switch to the Terminal tab, leaving the user on a blank or unrelated view.

- **Root cause 1**: The `listSelection` binding setter called `projectStore.activateProject(id:)` but never set `navigationStore.activeTab = .terminal`.
- **Root cause 2**: `PaneStatusRow` had its own `.onTapGesture` that intercepted clicks before the List's selection mechanism could fire. The tap handler called `workspace.focusPane()` without switching the tab.
- **Fix**: Added `navigationStore.activeTab = .terminal` in the `listSelection` setter. Simplified `PaneStatusRow` to a display-only component (removed `.onTapGesture`, hover effect, and `.contentShape`), so clicks bubble up to the List's selection binding and the entire branch section acts as a single click target.

### Terminal — Project Switch Destroyed Terminal Surface

Switching between projects (B→A→B) caused terminal B to restart with a fresh shell, losing all command history and output.

- **Root cause**: SwiftUI's `ForEach` with `@Observable` may dismantle `NSViewRepresentable` wrappers during re-evaluation. The `TerminalScrollView` was deallocated, triggering `NSView._finalize → removeFromSuperviewWithoutNeedingDisplay` on the child `TerminalNSView`, which freed the ghostty surface.
- **Fix**: `GhosttyAppManager` now holds strong references to `TerminalNSView` instances (`retainedTerminalViews`). `removeFromSuperview` no longer frees the surface. `TerminalPanel.makeNSView` reuses existing views. Explicit cleanup via `destroyPane()` when tabs are actually closed.

## New Features

### Terminal — Command Finish Notifications

When a long-running command (>5 seconds) completes in a non-focused terminal pane, the app now notifies the user. Uses ghostty's native OSC 133 shell integration — zero polling overhead.

- **Sound**: Plays configurable alert sound when the pane is not focused
- **Dock bounce**: `NSApp.requestUserAttention` when the app is in the background
- **System notification**: `UNUserNotificationCenter` push when the app is in the background
- **Config**: `notify-on-command-finish = unfocused`, `notify-on-command-finish-action = bell`, `notify-on-command-finish-after = 5`

### Settings — Configurable Alert Sound

New "Notifications" section in Settings (Cmd+,) with an Alert Sound picker. 14 macOS system sounds (Basso, Blow, Bottle, Frog, Funk, Glass, Hero, Morse, Ping, Pop, Purr, Sosumi, Submarine, Tink) + None. Preview plays on selection. Persisted via `UserDefaults`.

### Terminal — Zero Padding for Split Panes

Ghostty's default 2px window padding created visible gaps between split panes. Now set to 0 by default via `GhosttyConfig.loadDefaults`. Users can override in their config file.

## Internal Improvements

### Ghostty Action Callbacks — Synchronous Dispatch

All `handleAction` callbacks changed from `DispatchQueue.main.async` to synchronous calls. `tick()` already runs on the main thread; the async dispatch caused macOS to delay delivery for backgrounded apps, breaking notifications and bell events.

### Background Tick Timer

Added a 0.5s timer (`startBackgroundTick`/`stopBackgroundTick`) that calls `ghostty_app_tick` when the app is inactive. Ensures terminal events (bell, title changes) are processed even when the app is in the background.
