# openOwl v1.0.8 Release Notes

**Release Date**: 2026-05-10

This release lands the file-explorer follow-up work that started from a "click does nothing on a 1.4 MB `.dic` file" report and grew into a full revisit of how openOwl handles non-trivial file sizes, plus per-click memory leak hardening, sidebar polish, and a sidebar/terminal module split.

---

## Critical Fixes

### File explorer silently rejected files over 1 MB

A 1 MB hard cap inside `openFileInTab` returned early without any UI feedback, so clicking a 1.4 MB hunspell dictionary, a slightly large JSON, or a moderate log file looked like the editor had broken. The user had no way to tell openOwl was refusing the file on purpose.

**Fix:** Replace the silent cap with a three-tier model that always shows something:

| Tier | Range | Behavior |
|---|---|---|
| Normal | < 10 MB | Full features — syntax highlighting, tree-sitter, edit |
| Large | 10–50 MB | Auto-enters read-only mode with `CodeLanguage.default` (no tree-sitter), orange banner with **Enable Anyway** button |
| Huge | ≥ 50 MB | `NSAlert` confirmation before opening, then enters Large mode |

Image files keep a separate 50 MB cap because `NSImage` decode cost is unbounded.

### Right-click on a file auto-loaded it into the editor

`OutlineTreeViewController.menuNeedsUpdate` programmatically selected the right-clicked row to drive context-menu actions, but didn't suppress the resulting `selectionDidChange` notification — the file got loaded into a new tab as a side effect of opening the right-click menu, even when the user only wanted **Reveal in Finder**. Multiple right-clicks could pin many `NSTextStorage` instances in `tabStorages`, looking like a memory leak.

**Fix:** Wrap the programmatic `selectRowIndexes` call in `isProgrammaticSelection = true / false` so the existing guard in `outlineViewSelectionDidChange` short-circuits.

### Two huge-file dialogs stacked on a single click

`NSAlert.runModal()` runs a nested event loop on the main thread. SwiftUI's `.onChange(of: selectedNodeID)` Task got scheduled inside that nested loop and ran `openFileInTab` again before the first dialog returned, producing two stacked confirmation dialogs for a single file click.

**Fix:** New `hugeFilePending: Set<URL>` state guards re-entrant calls during the modal.

### Worktree branch rename was UI-only

`Sidebar > worktree row > Rename Branch` mutated the in-memory display name and persisted it, but never called `git branch -m`. The sidebar showed the new name while `git branch --show-current` reported the old one — sometimes for hours before the user noticed.

**Fix:** `ProjectStore.renameWorktreeProject` is now `async throws` and runs `git branch -m` against the worktree's own checkout before touching UI state. Failures (name already exists, invalid characters, etc.) surface via a per-row `.alert`.

## New Features

### Loading / closing progress overlay for large-mode tabs

Opening or closing a Large/Huge tab triggers `NSLayoutManager` to build or tear down line layout for the entire buffer on the main thread. For a 60 MB plain-text file this can block for 10+ seconds, so the UI looked frozen.

A centered `ProgressView` overlay ("Loading large-test.txt…" / "Closing large-test.txt…") now paints **before** the synchronous freeze (a 50 ms `Task.sleep` in the open path and `DispatchQueue.main.async` in the close path force one SwiftUI render cycle to complete first), so users get clear "app is busy" feedback instead of "app crashed".

### Sidebar drag-to-reorder for active projects

The sidebar's `PROJECTS` section gets a SwiftUI `.onMove` on the active root projects `ForEach`. Long-press a project row, drag to a new position; worktrees follow their parent. Order persists to `~/.openowl/openowl.json`.

Backed by a new `ProjectStore.reorderRootProjects(orderedIDs:)` API that handles arbitrary id sequences, so it's easy to extend to inactive projects later.

### Free-terminal pane working directory tracking

Standalone terminal panes now track their working directory via `GHOSTTY_ACTION_PWD` (OSC 7 / shell-integration `cd` hook). A new `onContextDidChange` callback fires whenever the active pane's pwd shifts, letting the host refresh file explorer / git changes panels accordingly.

### Per-click memory leak hardening

Three uncapped caches across the file explorer got LRU eviction:

- `tabStorages` / `tabImageCache` cap at **10** open tabs; oldest non-active, non-dirty tab evicted on overflow.
- `projectScanCache` caps at **5** project entries; oldest project's `nodeIndex` (which can be tens of MB on large monorepos) evicted on overflow.

These were not real leaks — just unbounded growth that masqueraded as one across long sessions.

## Known Limitations

- 60+ MB plain-text files still take 10+ seconds to layout via `NSLayoutManager` even in Large mode (no tree-sitter). This is an upstream `CodeEditSourceEditor` / TextKit limitation; only `mmap`-based virtualized scrolling (out of scope for 1.0.8) would fix it.
- Sidebar drag-reorder is active-projects only. Inactive projects keep their relative order. The underlying `reorderRootProjects(orderedIDs:)` API supports both, so this can be loosened later.
