# v1.0.2 三连修：Deployment CPU、侧边栏导航、终端拖拽

> 日期：2026-03-18 | 影响：100% CPU / 空白页导航 / 文件拖入错误终端 | 修复轮次：各 1 轮

---

## 1. 概述

v1.0.2 修复了三个独立 bug，其中两个（侧边栏导航、终端拖拽）属于同一类根因：**SwiftUI + AppKit 混合架构中，AppKit 子系统不尊重 SwiftUI 的视图状态**。第三个（Deployment CPU）是经典的 Foundation `readabilityHandler` EOF 自旋 + 无节流 UI 更新。

## 3. Bug 1：Deployment 100% CPU

### 问题

项目持续输出日志时（如 Node dev server），app CPU 飙升到 100%。

### 根因（三层叠加）

1. **Foundation `readabilityHandler` EOF 自旋**：管道 EOF 后 `availableData` 返回空 Data，handler 只 `return` 但没 `nil` 掉自身 → 系统立即重新调用 → 忙循环
2. **每 chunk 都触发 `@Observable` 变更**：`appendLog` 直接修改 `logContent` → 每秒数百次 SwiftUI 重渲染
3. **双重日志更新**：实时 `onOutput` 回调 + 0.5s 文件轮询 Timer 同时向 `logContent` 写入相同数据

### 解决方案

1. EOF 时 `handle.readabilityHandler = nil`
2. `logBuffer` + 200ms 合并 Timer → `logContent` 更新降到 ~5 次/秒
3. `activeStreamIDs` 追踪活跃流，有流时跳过文件轮询
4. 移除 `withAnimation` 的自动滚动

### 教训

- `readabilityHandler` + `guard !data.isEmpty else { return }` 是 Foundation 的经典陷阱。**必须在 EOF 时 nil 掉 handler**。
- `@Observable` 属性的高频变更 = 高频 SwiftUI 重渲染。对日志类场景，必须有节流层。

## 4. Bug 2：侧边栏点击分支跳空白页

### 问题

在 Git Changes 或 Deployments tab 下，点击侧边栏的分支行或分屏状态行，页面变成空白。

### 根因（两层）

1. `listSelection` setter 只调 `projectStore.activateProject(id:)` 但没设 `navigationStore.activeTab = .terminal` → 停留在当前 tab，内容为空
2. `PaneStatusRow` 的 `.onTapGesture` 拦截了 List selection → 焦点转到 pane 但 tab 不切换

### 解决方案

1. setter 中加 `navigationStore.activeTab = .terminal`
2. `PaneStatusRow` 去掉 `.onTapGesture`，变成纯展示组件，点击冒泡到 List selection

### 教训

- SwiftUI List 的 selection 是**隐式**的（通过 `.tag()` 匹配），**显式 `.onTapGesture` 会优先消费事件并阻断 selection**。在 List 子视图中慎用 `.onTapGesture`。

## 5. Bug 3：文件拖入错误终端（重复踩坑）

### 问题

从 Finder 拖文件到终端，路径出现在**上一个项目**的终端里，当前终端变空白。

### 根因

切换项目 tab 时，旧终端 NSView 通过 `.opacity(0)` 隐藏但**仍在 AppKit 视图层级中**。AppKit 的拖拽系统**不检查 opacity** → 拖拽事件被路由到不可见的旧终端 → `performDragOperation` 写入了错误的 surface 并通过 `window?.makeFirstResponder(self)` 抢走焦点。

### 解决方案

`TerminalScrollView` 的四个拖拽方法（`draggingEntered`、`draggingUpdated`、`prepareForDragOperation`、`performDragOperation`）加入 `isEffectivelyVisible` 检查，遍历 superview 链检测 `isHidden` 或 `alphaValue < 0.01`。不可见时拒绝拖拽，让 AppKit 路由到可见的终端。

### 教训

**这是本项目第三次因为 "AppKit 不尊重 SwiftUI 视图状态" 踩坑：**

| 日期 | 问题 | AppKit 不尊重的 SwiftUI 状态 |
|------|------|---------------------------|
| 2026-03-18 (搜索) | `.contentShape(Rectangle())` 阻挡 NSView 事件 | `.allowsHitTesting` |
| 2026-03-18 (拖拽) | 拖拽事件发到 opacity=0 的 NSView | `.opacity(0)` |
| 2026-03-18 (侧边栏) | `.onTapGesture` 阻断 List selection | List implicit selection |

**规则**：在 SwiftUI + AppKit 混合应用中，任何通过 `.opacity(0)` "隐藏"的 NSView **仍然能接收 AppKit 事件**（拖拽、鼠标、键盘）。必须在 AppKit 层面也做可见性检查。
