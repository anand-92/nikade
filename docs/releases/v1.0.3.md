# openOwl v1.0.3 Release Notes

**Release Date**: 2026-03-19

Performance + stability release: fixes WindowServer crash from Metal surface accumulation, adds project tab management, and improves deployment health check efficiency.

---

## Critical Fix

### WindowServer Crash from Metal Surface Accumulation

The app could freeze the entire macOS display (WindowServer) when multiple projects were open. Each terminal pane creates a Metal rendering surface backed by a CAMetalLayer. Previously, all surfaces rendered continuously — even those hidden via `opacity(0)` in inactive project tabs.

With 8 projects open, this meant 8+ Metal surfaces simultaneously submitting drawables to WindowServer (~45MB GPU memory each, ~480 compositor operations/second), eventually exhausting WindowServer's capacity.

**Root causes identified and fixed:**

1. **All activated tabs kept alive with `opacity(0)`** — Changed to only mount the active tab's terminal views in the SwiftUI hierarchy. Inactive tabs' surfaces are retained (shell processes stay alive) but removed from the view tree, so their CAMetalLayers are detached and stop rendering.

2. **`metalLayer.isHidden` visibility pipeline** — Added a complete visibility signal chain from `ContentView` through to `TerminalNSView`. When a terminal is not visible (inactive project, non-terminal tab, maximized sibling), its Metal layer is hidden so WindowServer skips compositing entirely.

3. **`@Observable` cascade creating phantom surfaces** — Internal bookkeeping dictionaries on `GhosttyAppManager` were tracked by Swift's Observation framework. Any surface registration triggered SwiftUI re-evaluation, which recreated `NSViewRepresentable` wrappers, which created MORE surfaces. Fixed by marking all internal maps as `@ObservationIgnored`.

4. **Scroll view reparenting thrash** — Each SwiftUI re-evaluation created a new `TerminalScrollView`, causing the `TerminalNSView` to be reparented (viewDidMoveToWindow nil→window cycle). Fixed by retaining and reusing scroll views in `GhosttyAppManager`.

**Result:** GPU memory usage dropped from ~360MB (8 surfaces active) to ~45MB (1 active). WindowServer CPU from ~30% to <5%. Verified via built-in diagnostic: `Metal 1/15` (1 active out of 15 total surfaces).

## Improvements

### Project Tab Close & Reorder

- **Close button**: Hover over a project tab to reveal an ✕ button. Clicking removes the project from the tab bar and JSON store. Files on disk are not affected. Re-adding the project later restores it.
- **Drag to reorder**: Drag project tabs to rearrange their order. Worktrees follow their parent project. Order is persisted.

### Auto-Discover Git Worktrees

When adding a project, openOwl now automatically scans `git worktree list` and adds any existing worktrees that aren't already tracked. Previously, only explicitly created worktrees were shown.

### Health Check Exponential Backoff

Deployment health checks that fail consecutively now back off from 30s to 5 minutes (cap). This prevents console spam and wasted TCP connections when a monitored service is down. Polling resumes at 30s automatically when the service recovers.

### Debug Diagnostics

- **⌘⇧I**: Copies a diagnostic snapshot to clipboard (DEBUG builds only). Includes Metal surface stats (active/paused/detached), scroll view count, and memory usage.
- **Status bar indicator**: Shows `Metal X/Y` (active/total surfaces) with a green/orange dot. Only in DEBUG builds.

## Dependency Changes

- **CodeEditSourceEditor**: Switched from upstream `CodeEditApp/CodeEditSourceEditor@0.15.2` to fork `sanvibyfish/CodeEditSourceEditor@fix/minimap-brightness-catalog-color` to fix `MinimapView.setTheme()` crash when accessing `brightnessComponent` on catalog colors.
